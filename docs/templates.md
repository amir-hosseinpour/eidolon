# Eidolon — engagement templates

A template is the recipe for an engagement workspace. It declares which
substrate to provision, what VMs to boot, which secrets are required,
which decision-fork policies apply, and what the operator's empty
workspace and skill pack look like.

`engage start --template <name>` materializes a template into a live
engagement.

## Search order

Templates are discovered from two roots, in this order:

1. `$EIDOLON_HOME/templates/` — operator installs. First match wins.
2. `eidolon/templates/` — bundled in the package.

A directory only counts as a template if it contains a `template.yaml`.
Names must match the directory name.

## Layout

```
my-template/
  template.yaml          # required, schema below
  scripts/
    bootstrap.sh         # referenced by template.yaml under scripts:
    install_tools.sh
  skills/                # optional; copied to ~/.claude/skills/<eng>/
    README.md
    skill1/SKILL.md
  workspace_skeleton/    # optional; copied into the engagement workspace
    notes/
    decisions/
    findings/
    scope.md             # autogenerated by orchestrator if missing
    targets.md
    README.md
  README.md              # human-readable docs (not loaded by orchestrator)
```

`scripts/`, `skills/`, and `workspace_skeleton/` are all optional. If
present, they are referenced by the loader at engagement-open time.

## `template.yaml` schema

```yaml
name: web-app-pentest             # must match directory name; ^[a-z0-9][a-z0-9-]{1,63}$
version: "1.0"                    # ^\d+\.\d+(\.\d+)?$
description: >
  One-line summary of what this template is for.

substrate_support: [docker]       # any of: docker, proxmox, mac, windows
                                  # mac/windows raise NotImplementedError in v0.1

network:
  name: "eng-{{engagement_id}}"   # rendered with the engagement id at provision time
  cidr: 10.43.0.0/24              # informational on Proxmox (VLAN allocated by substrate)

vms:
  - name: kali                    # ^[a-z0-9][a-z0-9-]{0,62}$, unique within template
    image: kalilinux/kali-rolling:latest    # docker image OR Proxmox template id (string)
    cpu: 4                        # 1..32
    memory_mb: 8192               # 128..131072
    privileged: false             # docker only
    capabilities: [NET_ADMIN, NET_RAW]      # docker only
    env:
      DEBIAN_FRONTEND: noninteractive
      LANG: C.UTF-8
    cmd: null                     # optional override; null = image default
    volumes: []                   # list of [host_path, container_path] tuples

secrets_required:                 # labels the operator must store before engage start
  - target_credentials            # see eidolon/orchestrator/lib/secrets.py for label format
  - burp_license

decision_forks:                   # per-fork-type defaults applied at engagement open
  - type: scope_edge              # one of: path_selection, mode_change,
    auto_resolve: false           # cred_disposition, noise_threshold, scope_edge
    default_message: >
      Operator review required before testing hosts not in declared scope.

scripts:                          # paths relative to template dir; must exist as files
  bootstrap: scripts/bootstrap.sh
  install_tools: scripts/install_tools.sh

workspace_skeleton: workspace_skeleton  # default; null to disable
skills: skills                          # default; null to disable
```

### Substrate notes

- `docker`: `image` is a Docker image tag. `vms[].capabilities` and
  `privileged` apply. `network.cidr` is honored — the substrate creates
  a per-engagement bridge with this subnet.
- `proxmox`: `image` is the **template VM id** to clone (operator must
  prepare offline). `network.cidr` is informational; the actual VLAN is
  allocated from `PROXMOX_VLAN_POOL_START..PROXMOX_VLAN_POOL_END`.
- `mac`, `windows`: declared support raises `NotImplementedError("v0.2")`
  in v0.1.

## Bundled templates

Three ship in v0.1.

### `blank-kali`

Bare Kali rolling container. 2 vCPU / 4 GB. Two fork policies
(`scope_edge`, `noise_threshold`). Use for ad-hoc work or as a starting
point for custom templates.

### `web-app-pentest`

Single Kali VM, 4 vCPU / 8 GB, `docker` substrate, with a curated
HTTP-focused toolchain (ffuf, nikto, sqlmap, wpscan, zap, nuclei). Four
fork policies (`scope_edge`, `noise_threshold`, `cred_disposition`,
`mode_change`). Optional secrets: `target_credentials`, `burp_license`.
This is the recommended starting point for web-app engagements.

### `ad-recon-single`

Single Kali VM on Proxmox cloned from a pre-built template (default id
9000). 4 vCPU / 8 GB, attached to a dedicated VLAN. Required secrets:
`ad_credentials`, `krb5_conf`. Operator must prepare the Kali template
once (impacket, nxc, bloodhound-python, kerbrute, certipy,
ldapdomaindump baked in).

## Authoring a custom template

1. Copy `eidolon/templates/blank-kali/` to
   `$EIDOLON_HOME/templates/<your-name>/`.
2. Edit `template.yaml`: rename, bump version, add `vms`, declare
   `decision_forks`, list any `secrets_required`.
3. Drop bootstrap scripts in `scripts/` and reference them.
4. (Optional) Customize `workspace_skeleton/` with starter notes,
   targets template, ROE checklist, etc.
5. (Optional) Add Claude Code skills under `skills/`. They are copied
   into the engagement-scoped skills directory at engage-start.
6. Validate: `eidolon templates info <your-name>` returns OK.

## Validation rules

- `template.yaml` `name` must match the parent directory name.
- VM names must be unique within a template.
- Every script referenced under `scripts:` must exist as a file.
- `workspace_skeleton:` and `skills:` must be directories if specified.
- Schema violations raise `TemplateValidationError(template_dir,
  reason)` and the loader refuses to expose the template.

Lib: `eidolon.orchestrator.lib.templates`.
