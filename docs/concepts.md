# Eidolon — concepts (v0.1)

The smallest set of mental models you need to operate the orchestrator. Each
concept maps to one or two Python modules under `eidolon/orchestrator/lib/`.

## Engagement

A bounded piece of offensive work: a slug, a purpose (`pentest` / `research` /
`ctf` / `training`), a scope document, a status (`active` / `closed` /
`erased`), and a workspace on disk. Identifiers look like
`ENG-<14 hex>` and never recycle.

The engagement is the unit of cleanup. `engage close` keeps the workspace and
revokes scope tokens; `engage erase` does that and marks the on-disk workspace
ready to nuke. The audit log carries forward across both.

Lib: `eidolon.orchestrator.lib.engagements`.

## Scope token

A short-lived JWT (HS256) bound to an engagement, a target list, an action
list, and a tier. Issued by `engage start` (initial token) or
`engage scope` (additional tokens for narrower work). The orchestrator
refuses any tool dispatch whose target/action/tier does not match the
verified token.

Tokens carry a `jti`. Revoke with `engage scope revoke --jti <jti>`. Once
the engagement closes, every issued token is revoked.

Lib: `eidolon.orchestrator.lib.scope`, `eidolon.orchestrator.lib.revocation`.

## Workspace

A per-engagement Markdown directory under
`$EIDOLON_HOME/engagements/<id>/workspace/`:

```
workspace/
  manifest.json       # template + scope provenance
  scope.md            # autogenerated, do not edit by hand
  notes/YYYY-MM-DD.md # operator + AI free-form notes
  decisions/<fork>.md # one file per resolved fork
  findings/<slug>.md  # structured findings
  log.jsonl           # append-only event log
```

Materialized from a template's `workspace_skeleton/` at engagement start.
The AI writes structured records via the `workspace_*` MCP tools or the
in-VM agent; the operator opens any file in `$EDITOR` directly.

Lib: `eidolon.orchestrator.lib.workspace`.

## Decision fork

A structured pause: the orchestrator (or AI) opens a fork to ask the
operator to choose. Five types exist (see
[`decision-forks.md`](decision-forks.md)).

- AI calls `fork_open` and waits.
- Operator's Claude Code subscribes to the SSE stream; new forks pop into chat.
- Operator answers; CC calls `fork_resolve`.
- Resolution lands in the workspace under `decisions/` and in the audit chain.

Lib: `eidolon.orchestrator.lib.forks`. Routes:
`/v1/engagements/{id}/forks` and `/v1/engagements/forks/{id}/resolve`.

## Secrets broker

A façade over three backends — `env`, macOS Keychain, 1Password CLI —
selected by `EIDOLON_SECRETS_BACKEND`. Templates declare the labels they
need (`secrets_required`); the operator stores values via
`eidolon secrets store <label>`; the broker resolves them just before
substrate provisioning.

Secret values never enter Claude Code chat. The MCP server exposes
`secret_present` (boolean) but not `secret_get`. In-VM tools fetch values
through the eidolon-agent socket.

Lib: `eidolon.orchestrator.lib.secrets`.

## Substrate

A pluggable backend for VMs and networks. v0.1 ships:

- `DockerSubstrate` — docker-py SDK, per-engagement bridge network,
  containers as "VMs". Good for web-app and CTF workflows.
- `ProxmoxSubstrate` — `proxmoxer` clones from a template VM, allocates a
  VLAN from the 1000-1999 pool, attaches a vmbr.
- `MacSubstrate`, `WindowsSubstrate` — stubs that raise
  `NotImplementedError("v0.2")`.

The ABC at `eidolon.orchestrator.lib.substrate.base` is small on purpose:
`provision`, `destroy`, `network_create`, `network_destroy`,
`secrets_inject`. Templates declare which substrates they support;
`engage start` picks the first one available on the host.

Lib: `eidolon.orchestrator.lib.substrate.*`.

## Templates

A directory under `eidolon/templates/` (bundled) or
`$EIDOLON_HOME/templates/` (operator). Each template is `template.yaml`,
`scripts/`, `skills/`, `workspace_skeleton/`, `README.md`. See
[`templates.md`](templates.md) for the schema.

Three ship in v0.1: `blank-kali`, `web-app-pentest`, `ad-recon-single`.

Lib: `eidolon.orchestrator.lib.templates`.

## Audit log

A SHA-256 hash-chained, day-segmented JSONL log under
`$EIDOLON_HOME/audit/`. Every engagement open/close, scope token mint or
revoke, fork open/resolve, secret access, and tool dispatch lands here.

Verify with `eidolon audit verify`. The chain is the source of truth for
"what did this engagement actually do."

Lib: `eidolon.orchestrator.lib.audit`.

## VM agent

A small Python daemon (`eidolon-agent`) baked into engagement-VM images.
Registers with the orchestrator on boot using `EIDOLON_VM_TOKEN`,
heartbeats every 60 seconds, and exposes a Unix-domain socket at
`/run/eidolon-agent.sock`. In-VM tools fetch secrets without ever
knowing the orchestrator URL or VM token directly.

Lib: `eidolon.vm_agent.main`.

## MCP server

`eidolon-mcp` is the bridge from Claude Code (or any MCP client) to the
orchestrator. It runs as a stdio process, holds the operator bearer
token, and exposes 19 tools across engagements, scope tokens, forks,
secrets, templates, and workspace. See [`api.md`](api.md) for the full
list.

Lib: `eidolon.mcp_servers.server`.
